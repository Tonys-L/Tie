# 项目概览

> AI 首次接触项目时必须阅读本节，理解业务上下文后再读代码。

## 系统定位

桌面级便签与提醒应用，以独立悬浮窗口形式在桌面上快速记录便签，支持时间提醒和多设备 Git 同步。单用户桌面工具，所有数据本地存储。

## 业务背景

用户需要在桌面上随手记录便签（类似纸质便利贴），并能为便签设置时间提醒。便签以独立透明窗口悬浮在桌面上，关闭窗口不退出应用（托盘常驻）。需要多设备间同步便签数据。

## 项目目标

- 便签管理：创建、编辑、置顶、调色、调透明度、归档/恢复、删除
- 提醒调度：为便签设置一次性或周期性提醒，到期弹窗 + 系统通知
- 数据同步：基于 Git 仓库的多设备数据同步
- 桌面常驻：托盘常驻、全局快捷键唤起、启动时恢复所有便签窗口

## 核心业务关系

| 模块 | 关联模块 | 影响方向 | 影响说明 |
|------|----------|----------|----------|
| Note | Reminder | Note→Reminder | 删除 Note 时级联删除关联 Reminder（DB 层 ON DELETE CASCADE + 代码层双保险） |
| Reminder | Note | Reminder→Note | 提醒触发时通过 note_id 打开对应便签窗口，冗余 note_title 用于通知显示 |
| Note | Window Manager | Note→Window | Note 的 window_state 字段决定窗口位置/大小，is_pinned 决定窗口置顶 |
| Git Sync | Note + Reminder | Sync→两者 | 同步时导出/导入两者数据，冲突按 updated_at 仲裁 |
| Reminder Scheduler | Note Window | Scheduler→Window | 提醒到期时后端直接创建便签窗口（URL 带 reminder 参数） |

## 项目阶段

开发中（MVP），核心功能已实现：便签 CRUD、提醒调度、Git 同步、托盘常驻。前端为原生 TypeScript + HTML（无框架），后端为 Rust + Tauri 2.0。

# 健康度检查

定期或在重大变更后，执行以下检查：

```text
□ constraints.md 中的不变量是否与代码一致
□ 禁止事项是否被违反
□ 架构状态是否需要更新（代码重构后架构状态可能升级）
□ 术语是否在其他文档中一致使用
□ 是否存在超过 30 天未更新但代码已变更的约束
□ 代码实现与约束描述是否一致
□ 模块定位表（如存在）是否与当前代码结构一致
```

发现不一致时，优先更新知识库，再继续开发。

# 文档联动规则

修改一个文档时，必须联动检查相关文档：

| 修改了 | 必须检查 |
|--------|----------|
| `constraints.md` | `glossary.md`、`boundaries.md`、`flows.md`、`lessons/` |
| `boundaries.md` | `constraints.md`、`flows.md` |
| `flows.md` | `constraints.md`、`boundaries.md`、本文件（核心业务关系） |
| `glossary.md` | `constraints.md` |
| 本文件（核心业务关系） | `boundaries.md`、`flows.md` |

联动检查不意味着必须修改，而是确认是否需要同步更新。
